ورشة عمل git commit و git push

  • مالذي قمنا به في الدرس السابق ؟

    قمنا باستخدام GitKraken كوسيط للعمل مع git… بمعنى أنه git client GUI.

  • لماذا Git ؟

    هو عبارة عن نظام مفتوح المصدر الهدف منه إدارة عمليات برمجة وتطوير الانظمة والمواقع.

    من خلاله يقوم الفريق البرمجي بمتابعة وإدارة النسخ للنظام الذي يعملون عليه. خلال العمل على أي نظام يقوم الفريق البرمجي بعمل تعديلات ونسخ من النظام.

    النسخة الأخيرة ليست دائمًا هي الأفضل قد يضطر أحيانًا المبرمج للرجوع إلى نسخ سابقة للعمل عليها أو تعديلها.

  • هل تتذكر هذا الموقف:

    كل مرة تقوم بعمل نسخة جديدة تقوم بتسميتها final حتى يصبح الأمر ممل

    أو تخيل أن هناك مطورين يرغبوا في التعاون والتشارك في نسخ الكود !

    وعليه… تأتي الحاجة لأنظمة إدارة الإصدارات Version Control Systems.

  • ماهو الهدف من git ؟

  • تنظيم النسخ من العمل.

  • إمكانية الرجوع للنسخة الأخيرة أو لنقاط معينة من نسخ سابقة، نعم نقاط سابقة من العمل كما لو كانت صور تم التقاطها على مراحل مختلفة من العمل ونريد العودة لاحداها حتى لو لم تكن الأخيرة… فقط بهذي السهولة.

  • ما معنى هذا الكلام ؟

    نحن هنا نتحدث عن ما يسمى ادارة النسخ Version Control.

    عند العمل على مشاريع برمجية كبيرة يشترك فيها فريق مطورين لتحديث والعمل على نفس الاجزاء من الاكواد، سواء أكان العمل مع مطور واحد آخر او العمل مع الاف المطورين في انظمة مفتوحة المصدر.

  • ماهي مهام التطوير الأساسية؟

    المهام الأساسية التي نقوم بها عادة كالتالي: انشاء ملفات جديدة … اضافة محتوى لها … حفظها … فتحها في حال التعديل … ثم لاحفظ مرة أخرى.
    تكمن المشكلة هنا في التخزين بعد التعديل … هنا يتطلب الموقف أحيانًا ذكر لماذا تم التعديل وما الاضافات او التعديلات التي حصلت ؟

    هذا الامر يبدو بسيطًا وسلسًا عند التعامل مع ملف واحد ومطور واحد لكنه كارثيًا في حال عدد كبير من الملفات وعدد من المطورين يعملون عليها هنا تتغير المعلوماتالتي نحتاجها إلى من قام بالتعديل ومتى قام بالتعديل ؟ وما هي العديلات التي حصلت ؟ ولماذا؟

  • هل يوجد نظام Version Control آخر؟

    نعم يوجد أنظمة أخرى:
    Mercurial, Bazaar و Darcs

  • كيف يعمل نظام version control ؟

    يوفر Git لنا طرقًا سريعة وحديثة لإدارة النسخ version control.

    توفر لنا مساراً تاريخيا يوضح معلومات كافية عن التحديثات.
    لدينا هنا عدة أشخاص يعملون على نفس الملفات بالطريقة التالية على مسار زمني timeline كما هو موضح:

    تخيل أنك تقوم بكتابة التاريخ باستعمال git !

  • ماهو التاريخ في حالتنا ؟

    التاريخ ببساطة هو repository خاصتك.

  • ماهي أحداث هذا التاريخ ؟

    كل حدث في repository يعرف بـ commit.

  • أين سأكتب هذا التاريخ ؟

    ستقوم بكتابته محلياً, على جهازك, وبإمكانك ارساله ليراه الآخرين على موقع GitHub. يسهل العمل التعاوني المشترك على الملفات حتى إذا كان العمل متزامن أي أن المطورين يعملون على ذات الملفات ووذات الوقت (تزامن).

    يستطيع المطور العمل على أنواع مختلفة من الملفات قد تكون تصاميم او ملفات برمجية أو وثائق او غير ذلك.

  • أين يتم حفظ هذا التاريخ ؟

    هل لاحظت مجلد .git

    هنا يتم الحفاظ على التاريخ كاملاً, لا تقم أبداً أبداً بالعبث بهذا المجلد.

  • ماهي الثلاثة الأوامر الرئيسية في git ؟

    1. لإنشاء المستودع الذي سيتم التخزين فيه repository:

      git init

    2. لعمل نسخة من الملفات الموجودة في المستودع repository إلى جهاز الكمبيوتر:

      git clone

    3. لمعرفة حالة repository:

      git status

  • كيف نقوم بعمل commit جديدة ؟

    الأمر التالي يجهز الملفات التي سيتم نسخها ليكونوا في قائمة الانتظار:

    git add .

    ثم الامر الذي يقوم بحفظ التعديلات، وهنا لابد من إضافة رسالة (وصف):

    git commit –m "Seperated HTML from CSS"

    أخيراً إرسال التعديلات إلى GitHub:

    git push origin master
